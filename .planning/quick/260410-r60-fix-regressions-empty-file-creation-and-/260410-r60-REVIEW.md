## Warnings

### WR-01: `cur` is re-assigned from `Pop()` before `writeErr` is checked — failed write silently pushes wrong item

**File:** `internal/app/app.go:59-66`

`Pop()` is called on line 59 and its return value overwrites `cur`. If `writeErr != nil` (line 64), the item pushed to `a.failed` on line 66 is the *next* item in the queue, not the item whose write failed. The item that failed to save is lost and will never be retried.

```go
// Current (buggy) — cur is overwritten before the error check
writeErr := a.writer.WriteLRC(song, cur.Filename, cur.Outdir)
cur, err = a.inputs.Pop()   // line 59: cur now points to the NEXT item
if err != nil { ... break }
if writeErr != nil {
    a.failed.Push(cur)      // line 66: pushes the NEXT item, not the failed one
}
```

**Fix:** Capture the item to retry before overwriting `cur`, or separate the Pop from the error check.

```go
writeErr := a.writer.WriteLRC(song, cur.Filename, cur.Outdir)
failed := cur                   // save before Pop overwrites cur
popped, err := a.inputs.Pop()
if err != nil {
    slog.Error("unexpected empty queue on pop", "error", err)
    break
}
cur = popped
if writeErr != nil {
    slog.Error("failed to save lyrics", "error", writeErr)
    a.failed.Push(failed)       // push the item that actually failed
}
```

### WR-02: `handleFailed` directly accesses `a.inputs.Queue` — bypasses queue abstraction and can corrupt state

**File:** `internal/app/app.go:110`

```go
a.failed.Queue = append(a.failed.Queue, a.inputs.Queue...)
```

`Queue` is an exported field, so this compiles, but the append shares the underlying array between `a.inputs.Queue` and `a.failed.Queue` when `a.inputs.Queue` has spare capacity. A future `Push()` on either queue could silently clobber the other's elements. This also bypasses queue methods entirely, meaning any future invariants added to `Push()` (e.g., deduplication, capacity limits) will not apply here.

**Fix:** Use `Push()` in a loop, which is already O(n) anyway:

```go
for !a.inputs.Empty() {
    item, _ := a.inputs.Pop() // Empty() guard means this cannot error
    a.failed.Push(item)
}
```

### IN-01: Custom `contains` helper in `writer_test.go` avoids `strings.Contains` for no gain

**File:** `internal/lyrics/writer_test.go:100-112`

The comment on line 100 says "to avoid importing strings in tests", but `strings` is part of the standard library and has zero binary-size cost for test binaries. The custom `findSubstr` loop is 10 lines of code that needs to be read and reasoned about when `strings.Contains` is a one-liner that every Go reader already knows.

**Fix:** Replace with `strings.Contains`:

```go
// In the test body (line 71):
if !strings.Contains(content, want) { ... }
```

Then remove the `contains` and `findSubstr` helpers entirely (lines 100-112) and add `"strings"` to the import block.

### IN-02: `queue.go` line 33 clears `Queue[0]` before reslicing — comment says "memory leak" but the concern is minor and the pattern is unconventional

**File:** `internal/app/queue.go:33`

```go
q.Queue[0] = models.Inputs{} // clear reference to avoid memory leak
q.Queue = q.Queue[1:]
```

The zeroing is a deliberate GC assist: after reslicing, `Queue[0]` in the original backing array is still reachable via the slice header until the array is collected or overwritten. For a queue of `models.Inputs` (small value types with no pointer fields except `string`), the GC pressure is negligible. The comment is accurate but may mislead future maintainers into thinking `models.Inputs` holds large heap objects. If `models.Inputs` grows pointer-heavy fields in the future this becomes more important, so the pattern is defensively correct — just worth acknowledging.

**Fix:** No change required. Optionally add more context to the comment:

```go
q.Queue[0] = models.Inputs{} // zero slot so strings in the old item can be GC'd
```
